**Audit:** NSCIM v1 cold audit, 2026-05-05

## Narrative

The intake pipeline starts cleanly — FS6000 file-syncs from a UNC share with retry-on-network-blip and stability-wait helpers, ASE pulls from SQL Server with batched dedup, and `ContainerScanQueuePublisherService` enqueues each scan into `containerscanqueues`. From there `ContainerCompletenessService.CheckContainerCompletenessAsync` is supposed to drain the queue, perform the 6-layer match-correctness checks (port/fyco/clearance/regime/date) and write `ContainerCompletenessStatus` rows. **As of the probe window the pipeline is broken between the queue and the CCS service.** 1,533 queue items have been Pending for up to 10 days, all with `retrycount=0` and `processedat IS NULL`, while ingestion continues to enqueue normally. The CCS service is *running* — its 30-second loop reports `0 pending` repeatedly in `Data/Logs/nickscan-20260505.txt:42-49` — but a probe with `SET LOCAL app.tenant_id='1'` shows 1,533 rows visible. **The TenantConnectionInterceptor is not re-running on EF-Core's pooled connection re-use, so the service's queries fall through the RLS COALESCE to `'0'` and silently return zero rows.** This is finding 2.01 and is the highest-priority issue surfaced by the audit; it explains why the topology agent saw `AnalysisQueueEntries=0` despite 16 Ready AGs (same root cause), and likely explains a swathe of "stalled" symptoms downstream.

Several scan-side data-integrity issues compound the queue-stall. **FS6000 dedup is by container number alone** (cache `_existingContainersCache` is keyed on `ContainerNumber`), so a re-scan of the same container is silently dropped — see 532 FS6000 containers in the last 14 days lacking a CCS row, 1,191 FS6000 scans with no queue row at all. ASE has analogous misses (3,014 scans without queue rows). These predate the current pipeline and won't self-heal without a backfill (finding 2.05). **Comma-joined container numbers** still exist as 10 stuck CCS rows from before the 1.18.0 split-at-publish fix; new ASE scans split correctly but the historical comma rows are dead-weight (finding 2.09). **Image filenames are shared across distinct containers** in 778 cases — same JPEG bytes loaded once into Postgres `bytea` per scan, fanning out the storage footprint and creating the cross-record `DuplicateImage` flag whenever a multi-container scan event is parsed (finding 2.03).

**Cross-record validation has a different fault** (finding 2.04). All 134 `crossrecordscans` rows have `splitjobid=NULL` despite the Python splitter being healthy and processing other jobs (89 `image_split_jobs`, 331 `image_split_results` exist). The handoff write-back from `MultiContainerValidationService.SubmitSplitJobForCrossRecordAsync` to the `crossrecordscans.splitjobid` column is broken in some way — every CRS row is missing the link, including ones from today. Combined with finding 2.14 (`PostICUMSValidationService` only sees post-1.18.0 multi-originals), large parts of the multi-container detection-and-split pipeline are non-functional.

The new record-completeness layer (`RecordCompletenessStatus` + `RecordExpectedContainer`) and Pattern A grouping look healthy: 994 distinct `ContainerGroupKey` values across 2,700 records, with up to 8 sibling declarations per shared physical container (`MRSU6042566` example). `RecordReconciliationWorker` is event-driven and the `RecordCompletenessBuilder.Build` correctly emits `containerGroupKey` only when a single-container declaration shares its container with another declaration. The 30,475 `AwaitingScan` `RecordExpectedContainer` rows are a healthy backlog reflecting BOEs known to ICUMS but not yet scanned.

## Open questions

1. **Is the TenantConnectionInterceptor re-firing in production?** The probe environment reproduces the gap (test C of `IntakeProbe8` shows `setting=''` after first connection re-use), but it's worth instrumenting the interceptor with an `INFO` log on each `ConnectionOpenedAsync` to see actual production timing. There may be a configuration knob (`Pooling=false` on the DbContext, or an EF retry policy that bypasses the interceptor in retry path) that mitigates in production but doesn't show up in the probe. The CCS service log lines need a `DbConnection.OpenedAsync` trace.
2. **Are there CCS rows that DO get created today?** The probe sees `last_update=2026-05-04 22:09` — but that doesn't preclude a tiny percentage of cycles succeeding when a connection happens to be fresh. Need to check `Data/Logs/nickscan-20260505.txt` for any `Saved N new completeness records` log line.
3. **Is `OriginalScanRecord.OriginalContainerNumbers` always source-faithful for historical rows?** The 1.18.0 fix splits at publish but doesn't claim to retroactively re-derive original-record `derivedrecordcount`. Backfill scope hinges on this.
4. **Why are there 297 picnumbers with multi-container collisions in `fs6000scans`?** Some are legitimate multi-container scans (the FS6000 scanner can emit one event for two containers on a truck). But 4-container `picnumbers` (`23301FS01202604290022`) need verification — this could be a scanner glitch, OR the XML parser is fan-ing one event into multiple scans incorrectly.
5. **`ContainerStatusReconciliationService` is registered (per topology) but its source code wasn't deeply audited.** Likely the source of the 24 `Status='Complete' AND BOEDocumentId IS NULL` invariant violations (finding 2.15). Worth a focused trace.

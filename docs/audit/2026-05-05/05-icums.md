# 05 — ICUMS ingestion + queues + submission

**Audit:** NSCIM v1 cold audit, 2026-05-05
**Agent:** ICUMS data-integrity layer (ingest → queue → submit)
**Scope:** Read-only audit of every ICUMS-side data path: BOE field/label
ingestion, raw JSON storage, the cross-DB BOE↔CBR fan-out, the four queue
surfaces, the file-based Outbox, the submission paths (Outbox writer +
ICUMSSubmissionService), and the manual escape hatches (ManualBOERequest,
CMRRedownloadQueue, IcumDataTransferController). Out of scope: match-
correctness rules (Agent 3), AG state-machine on submission completion
(Agent 4), full RLS surface audit (Agent 7), logging discipline (Agent 8).
Surfaces flagged for those agents are noted inline.

> Probed live 2026-05-05 against the running services and the five Postgres
> DBs. Probe code at `C:\temp\nscim-probe\Icums*.cs` (subcommands `icums`,
> `icums2`, `icums3`, `icums4`, `icums5`). All queries ran with
> `SET LOCAL app.tenant_id = '1'`.

## Scope confirmation

This pass exhaustively documents (a) the BOE JSON ingestion pipeline (every
field-mapping site in `IcumJsonIngestionService.cs` + `IcumPipelineOrchestratorService.cs`),
(b) the typed-vs-blob persistence surface in `boedocuments`,
`icumcontainerdata`, `manifestitems`, (c) the four queue tables plus the
file-based Outbox, (d) the two parallel submission paths (one alive, one
dormant, one stub), (e) the `documenttype` tagging from 2026-05-04, (f) the
order-flip "fix" landed in `f4ec289` and where it was AND wasn't applied,
(g) the BT-vs-BOE conflation upstream (regime-80 IM count), (h) the manual
BOE request lifecycle, and (i) the half-state CMR backfill scope vs
regime-80 transit guard.

Out of scope and deferred:
- Whether existing CBR matches are *correct* (Agent 3).
- Whether AGs advance correctly when WorkflowStage flips to `Submitted`
  (Agent 4).
- A full RLS-policy audit of the ICUMS tables (Agent 7) — gaps surfaced inline.
- Logging discipline (Agent 8) — `ingestionlogs` "Started but never closed"
  flagged inline as a logging finding.

## Narrative

### Field-completeness verdict: **Mostly. Two structural drops + one paint-job inconsistency.**

Operator concern was: *"are all fields and labels been ingested properly"*. The honest answer is **mostly yes** at the JSON parse step (the `IcumJsonIngestionService` parses every documented Header / ContainerDetails / ManifestDetails / ManifestItem field, with case-variation aliases and root-level fallbacks), but **partially no** at the persistence step:

1. **Finding 5.01 (P0)** — eight columns silently dropped on UPDATE: `masterblnumber`, `unmappedfield1..20label/value`, `unmappedfieldscount`, `unmappedfieldsoverflow`, `hasingestionwarnings`, `ingestionwarnings`. The `Update` SQL in `IcumDownloadsRepository` enumerates ~40 columns but excludes these. Zero of 119,669 BOE rows have unmapped-field tracking populated — the entire safety net for "fields we didn't know about" is dark. MasterBlNumber is on only 17% of rows and **all 5,721 IsConsolidated=true rows have NULL MasterBlNumber** — the operator-visible cargo grouping symptom that 2.16.3 has been chasing.
2. **Finding 5.02 (P0)** — the on-demand DownloadQueue path's BOEDocument mapper has a different + buggy field mapping (assigns `MasterBlNumber` from the JSON to the `BlNumber` column on the row). Two ingestion paths, divergent code, the same DB destination. Hardcodes `ContainerQuantity = 1`.
3. **Finding 5.10 (P2)** — naming confusion: `ManifestDetails.MasterBlNumber` (the model class) is set from `BLNumber` JSON keys in the VIN flow, but from `MasterBLNumber` keys in the main flow. Two different things share a name.

### Submission posture: **Functional but fragile.**

There are **three ICUMS submission code paths**. Two are alive, one is a stub:

1. **Active**: `ImageAnalysisOrchestratorService.SubmitPayloadsToIcumsAsync` (line 2148) and its retry partner `RetryPendingIcumsSubmissionsAsync` (line 1692). They write per-container `ICUMS_*.json` files to `Data/ICUMS/Outbox/ICUMS/`, POST them to `https://esb.unipassghana.com:26004`, and on success move to `/Acknowledged/` (487 files there). LiveSubmitEnabled is **true** in DB systemsettings (overrides the false default in appsettings.json).
2. **Active but anemic**: the legacy `_*.json` writes at the Outbox root (260 files) — `analysissubmissions` rows are stamped `TestSaved` and never advance (5.05). These predate the per-container ICUMS shape; they're the analyst+audit-decision dump that an early architecture used. The newest is from **2026-04-21** — three weeks dead.
3. **Stub**: `ICUMSSubmissionService.SubmitToICUMSAsync` is a *simulation* (`SimulateICUMSSubmissionAsync` with `Random.Shared.NextDouble() < 0.1` failure injection — finding 5.04). Drains `icumssubmissionqueues` (which has 0 rows lifetime). Hosted, runs every 10 min, dormant.

The order-flip fix from `f4ec289` was **partially applied** (5.03): the retry path is correct (ARCHIVE-FIRST, MARK-SECOND with explicit comment), but the primary submission path still does DB UPDATE before File.Move at lines 2212-2222. The retry path's invariant only holds if the primary path never crashes between the UPDATE and the Move — which is exactly what the fix was supposed to prevent.

### Manual escape hatches: **Both broken.**

`ManualBOERequest` rows: 1,747 stuck at `Queued`, oldest 47 days old, last successful Completed transition was 2026-03-25 (5.06). `ICUMSDownloadQueue` (prod-side, plural): 55 stuck at `Pending`, oldest 47 days, zero retry attempts (5.07). When ICUMS misses a container and an operator hits the manual button, the request is silently lost. Compounded by 5.18: the IcumDataTransferController's manual `TriggerTransfer` button is also a no-op.

### Half-state CMR / regime-80: **Backfill scope held.**

Per memory `feedback_regime80_cmr_is_transit.md`, the half-state CMR backfill must scope to regimes 40/70/90, not 80. Probe (`icums/5`) confirms: of 1,706 CMR-upgraded-to-IM rows, 640 are regime 40, 55 are regime 70, 943 are regime 80, 68 are regime 90. The 943 regime-80 rows are the historical mass-upgrade from before the carve-out (CHANGELOG 2.16.0 records 943 historical mis-flips fixed). The 5,499 native regime-80 IM rows that ICUMS itself sends as IM (5.15) are a separate upstream issue we cannot fix from this side.

### Documenttype tagging: **99.99% correct, 16 stragglers.**

Backfill applied 2026-05-04 tagged 32,903 rows correctly. Today only 16 rows (15 regime-40, 1 regime-80) ingested 2026-05-05 06:46 are mis-tagged NULL despite known regime codes (5.14). Likely a one-off race during the column rollout.

### `analysisqueueentries` and BOE↔CBR integrity

CBR pointing to deleted BOE: 0 active, 1 inactive (5.01-adjacent). 116,598 BOEs with NO active CBR — that's expected (most BOEs are CMR pre-decs that never get matched). **CBR.icumsboeid is the FK column name** (not boedocumentid as one might assume from EF naming). `analysisqueueentries` cardinality = 0 + missing RLS is flagged in topology §B and Agent 04's domain.

## Open questions

1. **Why did `ManualBOESelectivityService` stop processing rows on 2026-03-25?** (5.06) The service is registered, but no row has Completed since then. Service config flag flip? Silent exception? Service is registered in ServiceConfiguration.cs but not visible in the recent changelog. A SYSTEM-account probe of the running orchestrator would surface this.
2. **What ARE the 1,881 IM rows with zero manifest items?** (5.17) Probe sample 10 — is RawJsonData `ManifestItems` empty, or did `SaveManifestItemsBulkAsync` fail silently on a duplicate-key path? Phase-3 follow-up.
3. **Operator policy for the 5,499 native regime-80 IM rows?** (5.15) Should NSCIM trust the upstream IM clearance type, or override to CMR-Transit? Operator decision.
4. **Are the 260 legacy at-root Outbox files reconciled with `/ICUMS/Acknowledged`'s 487 files?** They reference different schemas; if the legacy files match analysissubmissions rows that never got their status updated (5.05), there might be containers that were "submitted" from one path but never confirmed.
5. **`analysisqueueentries` RLS gap** is in topology §B and Agent 04's domain, but it bears repeating: queue-entry tables that drive operator assignment with no tenant scope are a phase-2 multi-tenancy time bomb. Flag for Agent 7.
6. **Should `IcumContainerData` table exist at all?** (5.11) It's a lossy projection of `BOEDocument`. The `IcumDataTransferService` and orchestrator both write it. The only consumer in the audit window's Grep is reporting/stats. Replacing with a view of `BOEDocument` (across DBs via FDW or a copy) might simplify the model.

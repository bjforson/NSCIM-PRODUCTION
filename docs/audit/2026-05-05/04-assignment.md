## Narrative

**The most important fact: the 2.16.1 fix is in dead code.** The `AssignmentWorker.cs` file holds the orphan-AG `Cancelled` transition (lines 824-833 and 1227-1229) and the `UpsertQueueEntryAsync` call (line 447). The orchestrator service that's actually hosted has neither. Recent commits to `AssignmentWorker` are no-ops at runtime; the 14 `Cancelled` AGs in production today were cancelled exclusively by the manual `OrphanAgSweep.cs` probe between 19:37-19:49 UTC — there is no evidence of the in-process code ever transitioning an AG to `Cancelled`.

Once you accept that, the topology mystery dissolves. **`analysisqueueentries=0` is correct.** The queue is rebuilt only by `ReconcileQueueAsync`, which itself only iterates `state='Active'` assignments. Right now there are 0 active assignments (lease churn keeps everything Expired/Released within 15 min). Therefore: 0 active → 0 queue entries → workbench fast-path empty → fast fallback kicks in via `ImageAnalysisController.cs:225`. `pg_stat_user_tables` shows 782 inserts and 782 deletes today — the table is exercised, just empties to 0 between cycles.

**The lease-cycle churn observed (134 assignments / 24h for pimage)** isn't a runaway loop in the way it first appears. The 4-hour window where pimage was logged in saw the orchestrator auto-assign against AGs that *were* Ready at the moment of assignment, then transitioned through Decision Agent (shadow mode — but historically lit up these AGs end-to-end since 4c4931c landed). All 8 mis-tagged-consolidated declarations completed today. The remaining 16 Ready AGs are 100% orphan per the predicate at `ReadyGroupsCacheService.cs:307-345` — `non_orphan_count=0, orphan_count=16`. The `ReadyGroupsCacheService` orphan-AG guard correctly excludes them, so the orchestrator is now functionally idle.

**The state machine (4.04) is the highest-leverage cleanup.** `AnalysisStatusValidator.IsValidTransition` returns `true` for any transition out of `Cancelled`, `AgentProcessing`, or `Archived` because they're not in the validator's dict — the fallback at line 51 says "allow unknown statuses for backward compatibility." Submission-workflow uses this validator to gate AG → `Completed`; that means a Cancelled AG could be re-flipped to Completed silently. Add the missing entries (each with `{}` or restrictive set) and either change the unknown-from fallback to `return false` or log a hard warning.

**The phantom-assignment / 2-min-heartbeat puzzle (4.09) is more subtle than it sounds.** The actual gate is 60 min (`MaxIdleMinutesForReadiness=60`), so an idle browser tab keeps the analyst "ready" for an hour, during which the orchestrator can issue up to ~24 assignments back-to-back if AGs are eligible. Combined with the orchestrator's lease-renew logic (15-min lease, recreate on expire), today's pattern of pimage's 60+ Cancelled and 68+ Expired assignments in 24h reflects exactly the orphan-AG re-issue cycle that 2.16.1 was meant to break — but didn't, because the fix landed in dead code. After the manual probe sweep at 19:37 UTC the loop stopped against those 10 specific declarations; the remaining 16 Ready AGs are correctly held out by the `GetReadyGroupsForRoleAsync` orphan-AG guard.

**`DecisionSideEffectsService` and queue purge (4.06).** Three of four call sites instantiate it with no `ReadyGroupsCacheService`, so when an analyst submits a decision through the controller, `RemoveQueueEntryAsync` no-ops. Queue stays in sync only via `ReconcileQueueAsync` 1-2 min later. Functionally tolerable today (queue is empty anyway), but the moment 4.02 is fixed and queue entries become real, the divergence window will surface as analyst workbenches showing already-decided assignments for up to 2 min.

## Open questions

1. **Should the orchestrator attempt assignment at all when `GetReadyGroupsForRoleAsync` returns 0?** Currently it logs "no ready groups" and returns. That's correct — but the `_lastValidationTime` and `CleanupExpiredUserReadinessAsync` paths still run every cycle. If the system stays empty for hours, the orchestrator does ~100 useless DB hits/hour. Consider an early-return at AutoAssignGroupsAsync if no roles produced any work.

2. **The 60-min readiness tolerance vs the 2-min SignalR heartbeat — design intent?** Comment at `ImageAnalysisOrchestratorService.cs:1352-1354` says it was extended "to rely more on database." If that's the design, 4.09 is a P3 not P2. Phase 3 to clarify with operator.

3. **`ZombieAnalysisGroupSweeperService` lifecycle when AG status set is broader.** Today only `AnalystCompleted` zombies get archived. With `Cancelled` and `Archived` now in the enum, are there shapes the sweeper should handle (e.g. `Cancelled` AGs aged > 30 days)?

4. **Cross-record scans / split jobs (4.13).** 134 detections, 0 splits. Why isn't `MultiContainerValidationService.SubmitSplitJobForCrossRecordAsync` firing? Phase 3 needs to trace one detection through C# logs.

5. **`ContainerGroupKey` (Pattern A) coverage.** `GetReadyGroupsForRoleAsync` uses CGK if present at line 118-128; the second-chance pass rescues groups whose primary CGK-keyed CCS join missed. What's the current Pattern A population? Need a cardinality probe (Phase 3).
